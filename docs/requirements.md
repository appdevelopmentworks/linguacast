# linguacast 要求定義

version: 0.1 / status: draft

---

## 1. 背景と目的

世界には無料で公開された良質な音声・動画コンテンツ（研究者のポッドキャスト、大学講義、カンファレンス、実務家の発信）が大量に存在するが、その多くは英語をはじめとする外国語であり、言語の壁のせいで日本語話者の多くがアクセスできていない。この情報格差につけ込む形で、本来なら一次情報を無料で得られるはずの領域について、高額な情報商材や「詐欺的スクール」が販売されている。

linguacast の目的は、**外国語の一次情報を日本語の音声として届けることで、ユーザーが高額商材に頼らず自分で学び、自走できる状態をつくる** ことである。

副次的な設計目標として、**ランニングコストゼロで使えること**（＝マシンさえあれば無料で完結）を重視する。非力なマシンしか持たないユーザーのために、クラウドAPIへのフォールバックを用意する。

## 2. ターゲットユーザー / ペルソナ

- **P1: 独学者（メイン）** — 海外の一次情報を学びたいが英語のリスニングに難がある。移動中に音声で学習したい。高いスクールに課金する前に自分で情報に触れたい。
- **P2: 非エンジニアの一般ユーザー** — マシンスペックが低い。技術的なセットアップは最小限にしたい。ワンクリックで動いてほしい。
- **P3: 開発者/上級者（Shin自身を含む）** — 高性能マシンを持ち、ローカルLLMで完全ローカル運用したい。モデルや音声キャラを細かく選びたい。

## 3. スコープ

### 対象（in scope）
- 単一の動画/音声URLからの、文字起こし → 和訳 → 音声化の一連処理
- ローカル完結の処理と、クラウドへのフォールバック
- 全訳モードと要約（ポッドキャスト化）モード
- 字幕（SRT）出力と、動画同期の吹き替え音声生成
- 生成音声のスマホへのQR配信
- 良質な一次情報源のプリセットチャンネル表示（設定で変更可能）
- Windows / macOS 両対応

### 対象外（out of scope）— v0.1
- リアルタイム同時通訳
- 複数動画の一括バッチ処理（将来拡張）
- 話者ごとの声の割り当て（話者分離は将来拡張）
- 動画の映像編集・字幕焼き込み以外の映像加工
- Web公開・再配布を前提とした共有機能（個人利用が前提。9章参照）

## 4. 機能要件（FR）

### FR-1 動画/音声の入力と取得
- URL（YouTube等）を入力として受け付ける。
- `yt-dlp` で音声ストリームを取得し、ローカルに保存する。
- 動画の長さ（数分〜10時間超）を検知し、後段の処理方式ルーティングに使う。
- タイトル・長さ・チャプターマーカー（あれば）を取得し保持する。

### FR-2 音声抽出
- 取得したメディアから、Whisper入力に適した音声（16kHz mono 等）を `ffmpeg` で抽出する。

### FR-3 文字起こし（STT）
- **主**: ローカル `faster-whisper`（`large-v3`）で原語のまま文字起こしする。
- VAD前処理で無音・音楽区間のハルシネーション（同一フレーズ反復）を抑制する。
- **フォールバック**: ローカルWhisperが利用不可（GPU非搭載・処理が非現実的に遅い等）の場合、クラウドSTTに切り替える。ユーザーは設定で強制切替も可能。
- セグメント単位のタイムスタンプを保持する（SRTと同期の基盤）。

### FR-4 翻訳（原語 → 日本語）
- **主**: ローカルLLMで和訳する。翻訳ノードは以下を選択可能:
  - `TranslateGemma`（翻訳特化。専用プロンプト形式で呼び出す）
  - `Qwen3.6`（汎用・日本語強い。トーン調整の指示が可能）
- **3段フォールバック**: `Ollama → LM Studio → OpenRouter（クラウド）`。各段をヘルスチェックし、利用可能な最上位を使う。解決された層をUIに表示する。
- **クラウド層は OpenRouter に統一（v0.1）**: OpenAI互換の単一ゲートウェイで、1つのAPIキーで OpenAI / Anthropic / Gemini 等の多数モデルにアクセスできる。プロバイダ別アダプタは実装せず、ユーザーはモデルslug（例 `anthropic/claude-...`, `openai/gpt-...`, `google/gemini-...`）を選ぶ。キーはOSキーチェーン/ユーザー設定に保存。
- 3段とも OpenAI互換インターフェースで呼べる（Ollamaは互換モード or `/api`、LM Studio・OpenRouterは互換）。base URL＋キーを差し替える統一クライアントで実装する。
- 長文はチャンク分割し、直前文脈と用語集を引き継いで一貫性を保つ。
- （任意・推奨）**ハイブリッド二段翻訳**: TranslateGemmaで下訳 → Qwen3.6で「ポッドキャストのナレーションとして自然な日本語」に推敲。

### FR-5 要約 / ポッドキャスト化（任意モード）
- 長編transcriptを要約し、日本語ポッドキャストの台本を生成する。
- 入力が長い場合の方式ルーティング: 大きなコンテキストに収まる場合は1パス要約、超長尺は階層要約（segment → section → 全体）。
- チャプターマーカーがあれば話題境界の分割ヒントに使う。
- 「理解・圧縮」と「台本生成」を分離した多段処理とする。
- **台本フォーマットは話者数で自動分岐**:
  - 話者が一人（講義・独白）→ **ナレーション単体**
  - 複数人（対談・インタビュー・パネル）→ **対話形式**（話者ターンを明示した台本）
- 話者数の判定は v0.1 では台本生成LLMがtranscriptの内容から推定する（往復のやりとりの有無等）。正確な話者帰属が要る場合は話者分離（WhisperX + pyannote）を将来拡張として追加。
- 対話形式では、話者ロールごとに異なるVOICEVOX音声を割り当てる（例: ホスト/ゲスト）。

### FR-6 字幕（SRT）出力
- 原語SRTと和訳SRTを出力する。
- 和訳SRTは原語SRTのタイムスタンプを維持し、テキストのみ日本語に差し替える。

### FR-7 吹き替えモード（動画同期）
- 和訳SRTのタイミングに同期した日本語音声トラックを生成する。
- セグメント単位同期を行う: 各セグメントの音声を該当タイムスタンプに配置し、尺のズレを補正する。
  - VOICEVOX `speedScale`（約0.8〜1.3にクランプ）＋ ffmpeg タイムストレッチ（`rubberband`）＋ 隣接無音への吸収を併用。
- 生成した日本語トラックを `ffmpeg` で元動画に多重化して出力する。
- （リップシンクは対象外。セグメント単位の同期に留める。）

### FR-8 音声合成（TTS）
- **主**: VOICEVOX（ローカル）で音声化する。
- **音声キャラクター選択**: VOICEVOXの `GET /speakers` から話者/スタイル一覧を取得し、ユーザーがUIで選択できる。
- **サーバ未起動警告**: TTS実行前にVOICEVOX ENGINE（:50021）をヘルスチェックし、未起動なら日本語で明確な警告を出し、起動を促す。パイプラインをサイレントに失敗させない。
- **フォールバック**: VOICEVOXが使えない/非力な環境向けに、クラウドTTSに切り替え可能。**v0.1のクラウドTTSは Google Cloud TTS（Neural2）に統一**（Googleの認証情報を設定）。
- 長文は文/フレーズ単位で合成し連結する。

### FR-9 QRダウンロード配信
- 生成した音声ファイルをローカルHTTPで配信し、そのURLをQRコード化してUIに表示する。
- 同一LAN上のスマホでQRを読み取り、ブラウザで即再生・ダウンロードできる。
- 配信サーバは `0.0.0.0` にバインドし、HTTP rangeリクエストに対応（スマホでシーク・ストリーム再生可能）。
- ダウンロードURLには推測不能かつ期限付きのトークンを用いる。
- （将来拡張: プライベートRSSフィード、Tailscaleによる外出先アクセス。architecture.md参照。）

### FR-10 設定・モデル選択UI
- STT/翻訳/TTS の各段について、使用エンジン・モデル・フォールバック方針を設定できる。
- クラウドプロバイダとAPIキーの登録。
- VOICEVOX音声キャラの選択。
- 出力モード（全訳 / 要約ポッドキャスト、字幕のみ / 吹き替え）の選択。
- 現在の解決済みバックエンド層（ローカル/クラウド）の表示。

### FR-11 プリセットチャンネル
- 良質な一次情報源のYouTubeチャンネルを、カテゴリ分けしたプリセットとしてUIに表示する。
- 初期プリセット（設定で変更可能）:
  - **AI**: Andrej Karpathy (`https://www.youtube.com/@AndrejKarpathy`) / DeepLearning.AI (`https://www.youtube.com/@Deeplearningai`)
  - **プログラミング / IT**: freeCodeCamp (`https://www.youtube.com/@freecodecamp`)
  - **投資**: Point72 (`https://www.youtube.com/@point72careers`) / The Master Investor Podcast (`https://www.youtube.com/@TheMasterInvestorPodcast`)
- 設定でプリセットの追加・削除・並べ替え・カテゴリ編集ができる。
- プリセットのチャンネルURLは個別動画ではないため、選択時に `yt-dlp` でそのチャンネルの最近の動画一覧を取得してユーザーに選ばせる（v0.1は最小構成: チャンネルを開く / 直近アップロード一覧の表示）。
- 目的（ミッション）: 「何を観ればいいか分からない」段階のユーザーに、詐欺的スクールではなく良質な無料一次情報への入口を最初から提示する。

### FR-12 UI / UX（FFmpeg-UI 準拠の導線）
- 既存の FFmpeg-UI（Next.js + Tauri 2、テンプレート/スニペット一覧UI）の操作感に寄せる。
- 画面上部に **URL入力ボックス** を置き、URLの入力・実行でダウンロード→翻訳パイプラインが開始する。
- URLボックスの **直下にプリセットチャンネルのリスト** を表示する。FFmpeg-UIのスニペット一覧と同じ視覚パターン（`▶ ラベル` 形式のグループ化リスト、カテゴリ見出し付き）とする。
- 1画面で完結するシンプルな導線を優先（入力 → 開始 → 進捗 → 生成物 / QR）。

## 5. 非機能要件（NFR）

- **NFR-1 コスト**: ローカル完結時のランニングコストはゼロ。クラウド利用時のみ従量課金。
- **NFR-2 オフラインファースト**: ネット接続はメディア取得とクラウドフォールバック時のみ必須。ローカル経路はオフラインで動く（メディア取得を除く）。
- **NFR-3 性能**: 高性能GPU（RTX 5090級）では数分動画をエンドツーエンド1分以内を目標。長尺（10h級）でも実用的な時間（数十分〜1時間程度）で完了。速度低下の主因はハードであり、`faster-whisper`＋常駐LLM＋GPU TTSで実用速度を確保する。
- **NFR-4 堅牢性 / 再開可能性**: 長尺処理は各段の中間成果物を保存し、失敗時に途中から再開できる。
- **NFR-5 プライバシー**: ローカル処理では音声・テキストが外部に出ない。配信はLAN内が基本。トークンで保護。
- **NFR-6 ユーザビリティ（P2向け）**: 非エンジニアでも最小手順で使える。依存（ffmpeg/yt-dlp/VOICEVOX）不足は起動時に検知し、分かりやすく案内する。
- **NFR-7 拡張性**: STT/翻訳/TTSはアダプタ差し替え可能な設計（PriceProvider的な抽象化）。新プロバイダ追加が容易。

## 6. フォールバック設計（サマリ）

| 段 | 主 | フォールバック順 |
|----|----|------------------|
| STT | ローカル faster-whisper | クラウドSTT |
| 翻訳/要約 | ローカルLLM | Ollama → LM Studio → OpenRouter（クラウド, OpenAI互換） |
| TTS | VOICEVOX（ローカル） | Google Cloud TTS（Neural2） |

各段は「ヘルスチェック → 利用可能な最上位を選択 → 解決層をUI表示」を共通の契約とする。ローカルエンジンの欠如は警告であってクラッシュではない。

## 7. モデル選定（未決事項 / 要ベンチマーク）

翻訳ノードの品質はプロダクトの体感価値そのものであり、モデル選定は実測で決める。

- **候補**: TranslateGemma 27B（翻訳特化・Gemma 3ベース） vs Qwen3.6-27B（汎用・日本語強い・新しいベース） vs Gemma 4 系 vs 日本語チューニング系（Swallow等）。
- **評価軸**: adequacy（意味保持）、fluency/naturalness（ポッドとして自然な日本語）、用語一貫性、専門語・慣用句の扱い。
- **方法**: 代表クリップ3〜5本に同一transcriptを投げ、LLM-judge（Claude/Gemini/Qwen3.7-Max）＋人手ブラインド評価。WMT系の一般ドメイン指標だけでは口語ポッドキャストの品質は測れないため自前evalが必須。
- **有力な折衷**: ハイブリッド二段翻訳（TranslateGemmaで下訳 → Qwenで自然化）。
- **VRAM実装ノート**: 5090（32GB）なら Qwen3.6-27B（Q4 ≈16GB）＋ TranslateGemma 12B（Q4 ≈7.5GB）を同時常駐可能。翻訳ノードを12Bにすればスワップ無しでハイブリッドが組める。まず27Bで品質を見て判断する。

→ この選定はSession 3の開始時タスクとして実施する（session-plan.md参照）。

## 8. 制約・前提

- **クロスプラットフォーム対応（Windows / macOS）。** スタックは FFmpeg-UI と同じ Next.js + Tauri 2 で、同アプリは既に Mac（Apple Silicon / Intel）と Windows を配布実績あり。パス・コマンドはOS非依存に保つ。
- **GPUアクセラレーションはOSで分岐**（STTアダプタで吸収、NFR-7）:
  - Windows/NVIDIA（開発機は RTX 5090, CUDA 12.8）: `faster-whisper`（CUDA）
  - macOS/Apple Silicon: `mlx-whisper`（MLX）または `whisper.cpp`（Metal）。CUDAは使用不可。
- ローカルLLM（Ollama）は Apple Silicon（Metal）でも動作し、Macでも完全ローカル運用が可能。VOICEVOX も Mac ビルドあり。
- 外部ランタイム依存: `ffmpeg`, `yt-dlp`, `VOICEVOX ENGINE`, `Ollama`（および任意で LM Studio）。
- ローカルLLMは Ollama/LM Studio 経由（OpenAI互換HTTP）で呼ぶ。

## 9. 法務・コンプライアンス留意

- linguacast は **個人の学習利用** を前提とする。他者の著作物であるYouTube動画から音声を取得し、翻訳・吹き替えした成果物を **公開・再配布** する行為は、原著作者の著作権・翻案権やYouTube利用規約に抵触しうる。
- v0.1では共有機能をLAN内の個人デバイス間（自分のスマホへ送る）に限定し、公開配信機能は実装しない。
- UI上に、生成物の取り扱いは利用者の責任である旨、および個人利用を推奨する旨を明示する。
- （ミッションの観点でも、目的は「一次情報へ本人がアクセスして学ぶこと」であり、翻訳物の二次配布ではない。）

## 10. 受け入れの目安（v0.1完成条件）

- 数分〜1時間程度の英語ポッドキャスト動画URLを入力して、日本語音声（全訳/要約）とSRTが生成できる。
- ローカル経路（faster-whisper + Ollama + VOICEVOX）で完全ローカルに完走する。
- 非力環境を想定し、各段のフォールバック（クラウド）が動作する。
- VOICEVOXの音声キャラを選択でき、未起動時に警告が出る。
- 生成音声をQRコードでスマホに送り、再生・ダウンロードできる。
- 吹き替えモードで、和訳音声を元動画に同期・多重化した出力が得られる。
